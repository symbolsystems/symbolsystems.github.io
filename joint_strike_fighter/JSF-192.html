All 'if...else if' constructs will contain either a final else clause or a comment indicating why a final else clause is not necessary [JSF-192-3]

DESCRIPTION 

"All 'if...else if' constructs will contain either a final 'else' clause
or a comment indicating why a final 'else' clause is not necessary.
This rule only applies when an 'if' statement is followed by one or more
'else if'."

The rule assumes a comment explains why the final 'else' is not necessary
if it is placed in the same line as the closing brace '}' of last 'else if'
and contains 'else' word.

See also: CODSTA-23, MISRA2004-14_10
SINCE 

v9.4
NOTES 

The final 'else' can be empty (the rule will not report a violation), but
for clarity it's better to add a comment which explains why the final
else can be empty.

Sometimes it's better to use the 'switch' statement instead of a series of
'else if'.
BENEFITS 

"Providing a final 'else' clause (or comment indicating why a final 'else'
clause is unnecessary) ensures all cases are handled in an else if series"
EXAMPLE 

enum Status { S1, S2, S3 };
void foo(int a, enum Status s)
{
    if (a < 0)
    {
    }
    else if (a > 3)   // Violation - no final else nor comment
    {
    }
    
    if (s == S1)
    {
    }
    else if (s == S2)
    {
    }
    else if (s == S3)   // Violation - no final else and comment not recognized
    {
    }
    // Final else not needed because all possible cases handled

    // OK: final else not needed because there is no 'else if'
    if(a < 0)
    {
    }
}
REPAIR 

enum Status { S1, S2, S3 };
void foo(int a, enum Status s)
{
    // Fix: final else added
    if (a < 0)
    {
    }
    else if (a > 3)   // OK - final else added
    {
    }
    else
    {
    }
    
    // Fix: comment explaining why final else is not needed placed
    // in correct line
    if (s == S1)
    {
    }
    else if (s == S2)
    {
    }
    else if (s == S3)   // OK - comment with 'else' word is in line with '}'
    {
    } // Final else not needed because all possible cases handled

    // Sometimes a 'switch' is a better replacement for a series of
    // 'if...else if'
    switch (s)
    {
        case S1:
            break;
        case S2:
            break;
        case S3:
            break;
    }
}
