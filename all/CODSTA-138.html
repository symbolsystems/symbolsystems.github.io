The result of a built-in assignment operator should not be used [CODSTA-138-3]

DESCRIPTION 

The rule reports a violation if a result of an assignment operator is used. The
assignment should be a separate (top-level) expression.

Both simple (=) and compound (*=, /=, %=, +=, -=, >>=, <<=, &=, Ë†=, |=)
operators are checked.

Only built-in assignment operators are checked - C++ overloaded operators
( for example operator= ) are not checked.

See also: CODSTA-102, MISRA-046_b, MISRA2004-12_2_a, MISRA2004-13_1, PB-02
SINCE 

v9.5
NOTES 

Violation is not reported in following cases:
- top-level comma operator:
  i = 0, j = 0, k = 0;       // OK
  x + (i = 0, j = 0, k = 0); // Violation
- top-level 'for' initialization and increment sections:
  for ( i = 0; ; i = 0 );    // OK
  for (; i = 0; );           // Violation
  for (;; i + (j = 0) );     // Violation
BENEFITS 

"The use of assignment operators, simple or compound, in combination with other
arithmetic operators is not recommended because:
- It can significantly impair the readability of the code;
- It introduces additional side effects into a statement making it more
  difficult to avoid the undefined behaviour" [MISRA C:2012 Rule 13.4]
EXAMPLE 

// Examples of non-compliant code:

int g;
int foo(int i)
{
  int j[ i = 10 ];      // Violation
  foo( i += 1 );        // Violation
  if( i < 0 || (i = g));// Violation
  while(i = 10);        // Violation
  g = i = 2;            // Violation
  sizeof(i <<= 2);      // Violation
  return i *= 2;        // Violation
}
REPAIR 

int g;
int foo(int i)
{
  // Usually the compliant code uses separate assignment statement:
  i = 2;
  g = i;
  if( i < 0 || g)
    i = g;
}
