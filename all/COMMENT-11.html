The character sequence // shall not be used within a C-style comment [COMMENT-11-3]

DESCRIPTION 

This rule reports a violation if "//" is embedded in "/* ... */" comment.

A C-style comment begins with /* and continues until the first */ is
encountered.

MISRA C:2012 standard says:
   If a comment starting sequence // occurs within a /* comment,
   is it quite likely to be caused by a missing */ comment ending sequence.

See also: MISRA2004-2_3, COMMENT-12
SINCE 

v9.5
NOTES 

MISRA C:2012 standard suggests using conditional compilation (e.g. #if or
   #ifdef) instead of comments to "comment out" sections of code.

C99 added "//" comments.

To check other combinations of embedded comments see MISRA2004-2_3 and
COMMENT-12 rules.

This rule will ignore instances which are URI specifications, such as:
http://foo.com/, ftp://zork.com, telnet://quux, mailto://fred@flintstones.net
BENEFITS 

Detects possible accidental ommision of "*/" end-comment marker.
EXAMPLE 

/* Violation - some comment, end-comment marker accidentally omitted

<new page>
// embedded comment suggests commented out code
Perform_Safety_Critical_Code();

/* another comment closes the first comment so there are no parse errors */
REPAIR 

/* review the code and decide if end-comment marked should be added */

/* alternatively, if the decide the comment/code is correct, you could
   remove/mangle the embedded comment marker:

<new page>
embedded comment marker removed
Perform_Safety_Critical_Code();

end of comment */

/* the suggested way to "comment out" code is to use conditional compilation: */
#if 0
some_code();
#endif
