A thread-specific storage should not be created by a thread [MISRA2012-RULE-22_20-1]

DESCRIPTION 
This rule detects cases when a thread-specific storage pointer is 
created inside of a thread.

For a list of recognized thread-creating functions, see the rule BD-TRS-THRTHR.

This rule recognizes the following creation functions for 
thread-specific storage pointers:

<threads.h> (C Standard Library)
    int tss_create( tss_t* tss_key, tss_dtor_t destructor )
    
<pthread.h> (POSIX)
    int pthread_key_create(pthread_key_t *key, void (*destructor)(void*));

<processthreadsapi.h> (Windows)
    DWORD TlsAlloc();

If any of these functions is called (directly or indirectly) from a function which itself
is a thread entry point, the rule will report a violation.
SINCE 
2023.1
SECURITY RELEVANCE 
Creating a thread-specific storage pointer inside of a thread can cause 
undefined behavior. It can cause a thread-specific storage pointer to be accessed
before it is created.
BENEFITS 
This rule helps you prevent threads from accessing thread-specific storage pointers 
with indeterminate state.
EXAMPLE 

#include <pthread.h>
#include <stdio.h>

static pthread_key_t key;

void *thread(void *ptr)
{
    pthread_key_create(&key, NULL); // VIOLATION
    return 0;
}

int main(int argc, char **argv)
{
    pthread_t thread1;
    pthread_create(&thread1, NULL, &thread, NULL);              
    return 0;
}
REPAIR 
To repair the code, create the thread-specific storage pointer outside of the created thread.

#include <pthread.h>
#include <stdio.h>

static pthread_key_t key;

void *thread(void *ptr)
{
    
    return 0;
}

int main(int argc, char **argv)
{
    pthread_t thread1;
    pthread_key_create(&key, NULL); // NO VIOLATION
    pthread_create(&thread1, NULL, &thread, NULL);              
    return 0;
}
