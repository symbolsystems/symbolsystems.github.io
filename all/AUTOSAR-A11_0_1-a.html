A non-POD type should be defined as class [AUTOSAR-A11_0_1-a-4]

DESCRIPTION 

"Types that are not POD types are supposed to be defined as class objects,
as a class specifier forces the type to provide private access control for all
its members by default. This is consistent with developer expectations, because
it is expected that a class has its invariant, interface and could provide
custom-defined constructors." [AUTOSAR A11-0-1]

This rule detects non-POD structs and unions types.
See also: OOP-48
SINCE 

v10.4
NOTES 

A non-POD type is a type that meets at least one of the following conditions:
- It is a class (classes are always non-POD).
- It is a struct/union that has one of the following:
  + a user-declared constructor
  + a private or protected non-static data member
  + a base class
  + a virtual function
  + a non-static member with a non-POD type (or an array of non-POD elements)
  + a non-static member with type reference
  + a user-declared copy assignment operator
  + a user-declared destructor
BENEFITS 

This rule helps you improve code readability and facilitates code
comprehension.
EXAMPLE 

struct S1 {                              // Violation
private:
    int i;
};

struct S2 {                              // Violation
    S2();
    int k;
};
REPAIR 

struct S {                               // OK
    int i;
};

class C {                                // OK
public:
    C();
private:
    int k;
};
