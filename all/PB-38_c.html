String literals with and without encoding prefixes should not be concatenated [PB-38_c-3]

DESCRIPTION 

"If both string literals have the same encoding-prefix, the resulting
concatenated string literal has that encoding-prefix. If one string-literal
has no encoding-prefix, it is treated as a string-literal of the same
encoding-prefix as the other operand. If a UTF-8 string literal token
is adjacent to a wide string literal token, the program is ill-formed.
Any other concatenations are conditionally-supported with
implementation-defined behavior." [ISO/IEC 14882:2011]

This rule detects when a string literal with a non-empty encoding prefix is
concatenated with a string literal with an empty prefix.

The rule can recognize the following encoding prefixes:
'u8', 'u', 'U', 'L'.
SINCE 

2023.2
BENEFITS 

This rule helps you avoid undefined behavior and potential developer confusion.
EXAMPLE 

char16_t array1[] = u"stringR" "stringL";         // Violation
wchar_t array1[] = "Hello" L"World";              // Violation
REPAIR 

char16_t array1[] = u"stringR" u"stringL";        // OK
wchar_t array1[] = L"Hello" L"World";             // OK
