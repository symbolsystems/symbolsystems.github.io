Operands of shift operators shall have an unsigned type [MISRACPP2023-7_0_4-b-2]

DESCRIPTION 

This rule reports when operands of shift operators (<<, >>) and
their compound assignment versions (<<=, >>=) do not have an unsigned
type.
SINCE 

2023.2
EXCEPTIONS 

The rule does not report on the signed left-hand-side operand if:
* it is a non-negative constant of type T (before integral promotion), and
* the right-hand-side operand is also a constant in the range
  [0, sizeof(T) * CHAR_BIT - 1], and
* the shift operation does not result in "overflow".

The rule does not report on the signed right-hand-side operand
if it is a constant in the range [0, sizeof(T) * CHAR_BIT - 1],
where T is the type of the left-hand-side operand
(before integral promotion).
BENEFITS 

This rule helps you avoid unexpected or undefined behavior.
The result of shift operations is only guaranteed to be defined
if the sign bit is not affected.
EXAMPLE 

void foo(signed char sc, unsigned char uc)
{
    sc >> 2U;       // Violation - left-hand-side operand is not unsigned
    1U >> sc;       // Violation - right-hand-side operand is not unsigned
    sc >>= (int)uc; // Violation x2 - neither of the operands is unsigned

    1 << 2; // Exception - both operands are constants matching the exception
            // requirements
}

enum E : signed long
{
    U,
    V,
    W
};

void bar(E e)
{
    e << 1u; // Violation - left-hand-side operand is not unsigned
}
REPAIR 

void foo(signed char sc, unsigned char uc)
{
    (unsigned)sc >> 2U;  // OK - both operands are unsigned
    1U >> uc;            // OK - both operands are unsigned
    uc >>= (unsigned)sc; // OK - both operands are unsigned
}

enum E : unsigned long
{
    U,
    V,
    W
};

void bar(E e)
{
    e << 1u; // OK - both operands are unsigned
}
